# A/V 同步完善实施计划

## 目标

参考 `音画同步完整设计方案.txt`，对现有 A/V 同步进行完善。核心改动：
1. 提升音频时钟精度（硬件时钟替代 PTS 估算）
2. 实现 `SYNC_WAIT` 精确等待（当前只是 break，未实际等待）
3. 自适应同步窗口
4. Seek 时重置同步状态
5. 启用调试日志

## 现状与差距分析

| 设计文档要求 | 现状 | 本次改动 |
|---|---|---|
| 硬件音频时钟 | `ao_get_clock()` 用 PTS 估算，`hw_clock.c` 编译但未使用 | 改进 `ao_get_clock()` 使用 WASAPI 硬件时钟 |
| `SYNC_WAIT` 精确等待 | `video_timer_cb` 中 `SYNC_WAIT` 只 break，不等待 | 用 `precise_sleep()` 实现精确等待 |
| 自适应同步窗口 | 固定 20ms | 根据 exclusive/shared 模式自动调整 |
| `AudioClock` 抽象层 | 不存在 | **暂不实现**（见下方说明） |
| `wo_clock` waveOut 回退 | 不存在 | **暂不实现**（见下方说明） |
| Seek 同步重置 | `sync_seek()` 只重置统计 | 增加 PTS 基准重置 |
| 调试日志 | `SYNC_DEBUG_ENABLED` 未定义 | build.bat 中添加 |

### 关于 AudioClock / wo_clock 的说明

设计文档的 `AudioClock` 抽象层 + `HWClock` + `WOClock` 是一个完整的音频输出+时钟方案，其中 `HWClock` 自己创建 `IAudioClient` 并管理音频输出。但当前 `ao_wasapi.c` 已经是完整的音频输出实现（ring buffer + 播放线程 + 重采样）。

**策略**：直接改进 `ao_wasapi.c` 的 `ao_get_clock()` 使用 WASAPI 硬件时钟，而非创建第二个 `IAudioClient`。这更简单且避免设备冲突。`AudioClock` 抽象层可在后续版本中作为重构目标。

## 实施步骤

### Step 1: 改进 `ao_get_clock()` — 硬件时钟

**文件**: `src/audio_out/ao_wasapi.c`

当前 `ao_get_clock()` (line 384) 用 PTS 估算：
```c
clock = last_write_pts - ring_avail / bytes_per_sec
```

**改为**：优先使用 WASAPI `IAudioClock::GetPosition()` + `GetCurrentPadding()`：
```c
double ao_get_clock(AudioOut* ao) {
    if (!ao) return 0;
    // 优先：硬件时钟（需要 PTS 基准已设置）
    if (ao->last_write_pts > 0 && ao->clock) {
        UINT64 pos = 0;
        if (SUCCEEDED(ao->clock->lpVtbl->GetPosition(ao->clock, &pos, NULL))) {
            // 已播放秒数（设备时间）
            double device_sec = (double)pos / ao->out_rate;
            // 硬件缓冲延迟
            UINT32 padding = 0;
            ao->client->lpVtbl->GetCurrentPadding(ao->client, &padding);
            double hw_delay = (double)padding / ao->out_rate;
            // PTS 基准 = last_write_pts - ring 缓冲延迟
            // 当前播放 PTS ≈ last_write_pts - (ring缓冲 + 硬件缓冲)
            double ring_delay = (double)ring_avail(ao) / ao->bytes_per_sec;
            double clk = ao->last_write_pts - ring_delay - hw_delay;
            return clk > 0 ? clk : 0;
        }
    }
    // 备选：PTS 估算（原逻辑）
    if (ao->last_write_pts > 0 && ao->bytes_per_sec > 0) {
        double buffered_sec = (double)ring_avail(ao) / ao->bytes_per_sec;
        double clk = ao->last_write_pts - buffered_sec;
        return clk > 0 ? clk : 0;
    }
    return 0;
}
```

**关键**：`ring_avail` 反映 ring 中已写入但播放线程尚未消费的数据，`padding` 反映播放线程已提交给 WASAPI 但硬件尚未播放的数据。两者之和才是真正的缓冲延迟。

### Step 2: 实现 `SYNC_WAIT` 精确等待

**文件**: `src/core/player.c` — `video_timer_cb()` (line 121)

当前：
```c
if (sd.action == SYNC_WAIT) {
    break;  // 不等待，等下次 33ms tick
}
```

**改为**：离开临界区后精确等待，再重新获取时钟决策。
```c
if (sd.action == SYNC_WAIT) {
    int wait_ms = sd.wait_ms;
    LeaveCriticalSection(&p->vq_lock);
    if (wait_ms > 0 && wait_ms < 200) {
        precise_sleep(wait_ms / 1000.0);
    }
    // 等待后直接渲染（不再重新决策）
    EnterCriticalSection(&p->vq_lock);
    // ... 渲染逻辑
    break;
}
```

**注意**：不能在临界区内调用 `precise_sleep()`，否则会阻塞视频帧入队。

### Step 3: 自适应同步窗口

**文件**: `src/sync/sync.c`, `src/sync/sync.h`

添加 `sync_set_window()` 函数：
```c
void sync_set_window(SyncContext* ctx, double window_sec) {
    if (!ctx) return;
    ctx->sync_window = window_sec;
    ctx->drop_threshold = window_sec * 5;
}
```

**文件**: `src/core/player.c` — `player_open()` 中根据 WASAPI 模式设置窗口：
```c
if (ao_is_exclusive(p->ao)) {
    sync_set_window(p->sync, 0.010);  // ±10ms
} else {
    sync_set_window(p->sync, 0.020);  // ±20ms
}
```

**文件**: `src/audio_out/audio_out.h`, `src/audio_out/ao_wasapi.c`
添加 `ao_is_exclusive()` 查询函数。

### Step 4: Seek 时重置同步状态

**文件**: `src/core/player.c` — `player_seek()` (line 299)

在 seek 逻辑中添加：
```c
sync_seek(p->sync);  // 已有，重置统计
// 注意：ao_reset() 已经会清空 ring buffer，时钟会在下次写入时重新锚定
```

当前 `sync_seek()` 已重置统计。确保 seek 后 `ao_get_clock()` 在没有新音频数据时回退到墙钟（`elapsed_sec`），这在 `video_timer_cb` line 100-101 已处理。

### Step 5: 启用调试日志

**文件**: `build.bat`

添加 `/DSYNC_DEBUG_ENABLED` 编译选项（仅 debug 构建）：
```
cl /O2 /W4 /utf-8 /DSYNC_DEBUG_ENABLED ^
```
